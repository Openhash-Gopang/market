사용자 발화부터 L1 블록 생성, IndexedDB Hash Chain 갱신, Supabase PDV 백업까지 — 코드 기준 6단계 시퀀스 해설. gopang-app.js · gopang-wallet.js · main.pb.js · Worker v4.9 반영.

①

발화 → K-Market 실행

gopang.net · gopang-app.js · callAI() · _gwpLaunch()
사용자 입력
채팅창에 "짜장면 주문해줘" 입력
sendMessage()가 호출되어 callAI(text)로 진입합니다. history는 최근 20턴 (slice(-20,-1))으로 제한하고 DeepSeek KV Cache를 위해 캐시버스터 없이 SP를 전송합니다.
LLM 호출
callAI() → DeepSeek (SP-00, max_tokens 800)
SP-00은 gopang-app.js 내 CFG.system에 하드코딩된 문자열입니다. localStorage 오염 방지를 위해 system_base로 백업 후 매 호출마다 복원합니다. KV Cache 절감율 실측 96% (cached=1920/prompt=2002).
태그 감지
[GWP:kcommerce] 감지 → _gwpLaunch()
스트리밍 응답에서 [GWP:kcommerce] 태그를 감지하면 getService('kcommerce')로 서비스 정의를 조회한 뒤 _gwpLaunch()를 실행합니다. 기존 탭이 열려 있으면 닫고 새 ctx로 재시작합니다 (P2 수정).
탭 오픈
market.gopang.net 새 탭 실행
컨텍스트를 btoa(unescape(encodeURIComponent(ctx)))로 Base64 인코딩한 뒤 URL 파라미터로 전달합니다. 모바일 팝업 차단 우회를 위해 사용자 탭 직후 예약한 빈 탭(_preTab)의 URL을 교체합니다.
market.gopang.net/webapp.html ?gwp=1 &token={_USER.guid} &origin=https://gopang.net &ctx={base64} &ctx_enc=b64
②

판매자 검색 → profile 팝업

market/webapp.html · SP-KMARKET · search_entities RPC
SP 로드
SP-KMARKET GitHub raw fetch (캐시버스터 없음)
loadSystemPrompt().then(() => sendMessage()) 순서로 SP 로드 완료 후 ctx를 자동 전송합니다. 캐시버스터(?t=Date.now()) 제거로 DeepSeek KV Cache가 정상 적중합니다 (P1 수정).
검색
[SEARCH] 태그 감지 → /search RPC
SP 지시에 따라 LLM이 [SEARCH]{"keyword":"짜장면","occupation":"중식 음식점업"} 태그를 출력하면 Worker /search 엔드포인트를 통해 Supabase search_entities RPC를 호출합니다. p_entity_type: null로 설정해야 business 타입 업체가 검색됩니다 (버그 수정).
팝업
[OPEN_PROFILE] → profile.html 팝업
2차 LLM 호출에서 [OPEN_PROFILE:{"handle":"@geumneung","guid":"dummy-hanlim-003"}] 태그가 나오면 users.gopang.net/profile.html을 팝업으로 엽니다.
users.gopang.net/profile.html ?gwp=1 &guid=dummy-hanlim-003 &opener_origin=https://market.gopang.net
주의

params 선언이 반드시 OPENER_ORIGIN 선언보다 앞에 있어야 TDZ ReferenceError를 피할 수 있습니다. p_entity_type: 'org'를 사용하면 business 타입 업체가 누락됩니다.

③

GWP 서명 흐름 (3-탭 relay)

profile → market → gopang → market → profile
profile ──GWP_SIGN_REQUEST──▶ market ──relay──▶ gopang gopang ──GWP_SIGN_RESPONSE─▶ market ──relay──▶ profile ※ market이 중계자(relay) 역할. Same-Origin 정책상 profile↔gopang 직접 교신 불가.
주문 요청
주문하기 클릭 → requestSign() 즉시 호출
중간 확인 시트 없이 requestSign()이 직접 호출됩니다. tx 객체(seller_guid, outputs, items, total)를 담은 GWP_SIGN_REQUEST를 window.opener(market)에 postMessage합니다.
서명 UI
_handleGwpSignRequest() → 서명 확인 카드 렌더링
gopang 채팅창에 인라인으로 판매자명·품목·합계·현재잔액·결제 후 잔액을 보여주는 카드를 렌더링합니다. 잔액 부족 시 서명 버튼이 비활성화됩니다. 서명 중복 방지를 위해 _gwpSignPending[sessionId]에 tx와 sourceWin을 임시 저장합니다.
서명 실행
wallet.sign(tx) → Ed25519 서명
_gwpSignExecute()가 window.gopangWallet.sign(tx)를 호출합니다. 내부 흐름은 아래와 같습니다.
// gopang-wallet.js sign() 내부 1. buildPrevSettleHash() → IDB financial_state.block_hash 읽기 → null이면 최초 거래 (L1 prev 검증 건너뜀) 2. buildTxWithPrevHash({..., prevSettleHash}) → tx.input.prev_settle_hash = block_hash → tx.input.balance_claimed = bs-cash 3. signTx(privKey, tx) → tx_hash = sha256(sortedStringify(tx)) → buyer_sig = Ed25519(privKey, tx_hash) 4. return { tx, tx_hash, buyer_sig, buyer_public_key, prev_settle_hash }
응답 전달
GWP_SIGN_RESPONSE → market → profile
서명 결과를 market 탭(_gwpTab)으로 전송합니다. market의 _gwpSignResponseHandler가 이를 수신해 profile로 중계합니다. 핸들러 중복 등록 방지를 위해 _setupGwpSignRequest()로 전역 플래그를 관리합니다.
도메인 격리

gopang.net과 users.gopang.net은 IndexedDB가 완전히 분리됩니다. 서명은 반드시 gopang.net의 wallet에서 수행해야 합니다. profile.html에서 서명을 시도하면 별도 키가 생성되어 UNREGISTERED_KEY (403)가 발생합니다.

④

/biz/order → L1 4단계 검증

Worker v4.9 · L1 PocketBase 0.22.14 · main.pb.js
주문 POST
profile → Worker /biz/order → L1 /api/tx
profile이 GWP_SIGN_RESPONSE를 수신하면 즉시 Worker /biz/order에 POST합니다. Worker는 CORS·인증을 처리한 뒤 L1 /api/tx로 위임합니다.
// /biz/order POST body { tx, tx_hash, buyer_sig, buyer_public_key, from_guid: "2601:db80:...", seller_guid: "dummy-hanlim-003", l1_node: "KR-JEJU-JEJU-HANLIM", prev_settle_hash:"4a6760...", balance_claimed: 1000000, outputs: [{ recipient_guid, amount }, ...] }
L1 검증
4단계 파이프라인 (main.pb.js)
PocketBase 0.22.14 JS 런타임에서 "1=1" 필터와 IPv6 직접 삽입 시 패닉이 발생합니다 (P12 수정). 전체 로드 후 JS filter()로 비교합니다.
1단계 서명 형식 !/^[0-9a-f]{64}$/.test(tx_hash) → INVALID_SIGNATURE 2단계 공개키 등록 gdc_keys에서 buyer_public_key 조회 없으면 → UNREGISTERED_KEY (403) 3단계 prev_settle_hash 일치 최신 블록 content_hash와 비교 불일치 → STALE_STATE (409) 이중지불(동일 prev 재사용) → STALE_STATE (409) 블록 없음(최초 거래) → 건너뜀 4단계 잔액 확인 balance_claimed < sum(outputs) → INSUFFICIENT_BALANCE (402)
블록 생성
blocks 컬렉션 저장 + l1_ledger 앵커링
content_hash = sha256(tx_hash + buyer_sig + prevBlockHash) // prevBlockHash: 이전 블록의 content_hash, 최초이면 "GENESIS" blocks 컬렉션 저장: { block_type, tx_hash, buyer_guid, seller_guid, buyer_sig, outputs, prev_block_hash, content_hash, height, prev_settle_hash } L1 응답: { ok: true, block_hash, tx_hash, buyer_claim }
DB 앵커링
Worker → Supabase fs_ledger 3행 + pdv_log
L1 응답 수신 직후 Worker가 동기적으로 Supabase에 기록합니다.
// fs_ledger 3행 (동일 tx_id) 구매자: direction=debit, fs_account=pl-purchase 판매자: direction=credit, fs_account=pl-revenue 플랫폼: direction=debit, fs_account=pl-platform_fee // pdv_log openhash_anchored=true, via_worker=true, reporter_svc='kmarket'
⑤

GWP_DONE → wallet 자기갱신

profile → market → gopang · redeemClaim() · Hash Chain
L1 ──block_hash──▶ Worker ──▶ profile ──GWP_DONE──▶ market ──▶ gopang ※ 구멍 B (P11 수정): market gwpDoneHandler에 gopang 포워딩 1줄 추가.
GWP_DONE 전파
profile → market → gopang (포워딩 체인)
profile이 GWP_DONE을 market opener에 전송하고 300ms 후 자동 닫힙니다. market은 gwpDoneHandler에서 자체 처리 후 e.data 그대로 gopang으로 포워딩하고 500ms 후 닫힙니다. e.data 전체를 넘기는 것이 중요합니다 — 필드를 재구성하면 미래 필드 추가 시 누락 위험이 있습니다.
// GWP_DONE 페이로드 { type: 'GWP_DONE', block_hash: "3951191b...", // L1 content_hash tx_hash: "txhash...", buyer_claim: { direction:'debit', fs_account:'bs-cash', amount:6000 }, pdvData: { who, when, where, what, how, why }, reporter_svc:'kmarket' }
자기갱신
redeemClaim({block_hash, claims}) — gopang-wallet.js
gopang의 GWP_DONE 핸들러가 window.gopangWallet.redeemClaim()을 호출합니다. buyer_claim이 단일 객체로 오는 경우 [buyer_claim]으로 배열 변환합니다 (구멍 C 수정).
// redeemClaim() 내부 (gopang-wallet.js) 1. IDB financial_state 로드 fs = { 'bs-cash': 1000000, ... } 2. claims 적용 direction='debit' → fs['bs-cash'] -= 6000 → fs['bs-cash'] = 994000 3. IDB 저장 (핵심) financial_state = { state: { 'bs-cash': 994000 }, block_hash: "3951191b...", ← 갱신 updatedAt: "2026-06-12T..." } 4. Hash Chain 기록 h_i = sha256(h_{i-1} + tx_hash + block_hash + height) IDB hash_chain[height=3] = { local_hash, prev_local_hash, prev_settle_hash, new_settle_hash, block_hash, recorded_at }
불변 조건

거래 N 완료 후 거래 N+1이 성공하려면 반드시 IndexedDB financial_state.block_hash === L1 blocks 컬렉션 최신 content_hash여야 합니다. 이 조건이 깨지면 무조건 409 STALE_STATE가 발생합니다.

⑥

PDV 기록

_recordPDV() · recordPDV() · localStorage + Supabase
중복 판단
reporter_svc 확인 → 중복 방지
gopang의 GWP_DONE 핸들러는 msg.reporter_svc가 'kmarket'이면 자체 PDV 기록을 건너뜁니다 (원칙 P5). market이 이미 Worker 경유로 기록했기 때문입니다.
gopang 내부
_recordPDV() → localStorage + Supabase 직접
reporter_svc가 없을 때만 실행됩니다. localStorage에 최대 1000건을 캐시하고, Supabase pdv_log에 직접 POST합니다.
P17 수정 _recordPDV user_guid / device_fp fallback

_USER.guid는 Supabase user_profiles와 연결된 후에야 채워지는 필드입니다. 주문 완료 시점에는 _USER.ipv6만 존재하는 경우가 많아 user_guid: _USER.guid가 undefined가 되고 JSON 직렬화 시 키 자체가 생략되어 Supabase NOT NULL 제약에 의해 400 Bad Request가 발생했습니다.

// 수정 전 (오류 발생) user_guid: _USER.guid, // undefined → 키 생략 → 400 device_fp: _USER.fp, // undefined → null // 수정 후 (P17) const _effectiveGuid = _USER?.guid || _USER?.ipv6 || USER_GUID; user_guid: _effectiveGuid, device_fp: _USER?.fp || _USER?.fpHex?.slice(0, 32) || null, who_name: 'GUID:' + (_effectiveGuid?.slice(0, 8) ?? 'unknown'),
market 경유
recordPDV() → Worker /pdv/report (원칙 P2)
market의 recordPDV()는 반드시 Worker /pdv/report를 경유합니다. Worker가 Supabase pdv_log에 INSERT하면서 openhash_anchored=true, via_worker=true로 설정하고 OpenHash 앵커링을 수행합니다. 직접 INSERT는 원칙 P2 위반입니다.
// pdv_log 최종 레코드 { user_guid: "2601:db80:...", record_type: 'service_task', who_name: "GUID:2601db80", what: "짜장면 주문 완료", why: "K-Market 서비스 이용", block_hash: "3951191b...", openhash_anchored: true, via_worker: true, reporter_svc: 'kmarket', session_id: "6f156447-..." }
